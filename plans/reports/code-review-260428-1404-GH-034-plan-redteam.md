## Verdict

**ACCEPT WITH CHANGES.** Two critical bugs (one wrong type signature, one false claim about cache). Several major concerns around fail-open semantics, schema-mutation surprise, and trust-boundary edge cases. Plan is otherwise well-scoped, KISS-compliant, and respects the FR-035 sibling pattern. ## Critical Issues (must fix before implementation)

### C1. `response_body_filter` return type is wrong

- **Where:** `phase-02-gateway-wiring.md` §4 ("response_body_filter Hook (NEW)")
- **Problem:** The plan declares
  ```rust
  ) -> pingora_core::Result<Option<std::time::Duration>>
  ```
  This is wrong twice over. (a) Researcher-02 §1 explicitly documents the trait method as returning `pingora_core::Result<()>`. (b) The existing `request_body_filter` already in `crates/gateway/src/proxy.rs:284-290` returns `pingora_core::Result<()>` — same family of hook on the same Pingora 0.8 (Cargo.toml line 28 confirms). Body of the hook ends with `return Ok(None)` / `Ok(None)` in three places, which is also wrong. Implementation will fail to compile or — worse — the `async-trait` macro will silently shadow the default trait impl with a wrong-typed method that's never called, and the redactor will never run.
- **Suggested fix:** Change return type to `pingora_core::Result<()>` everywhere in Phase 02; replace every `Ok(None)` with `Ok(())`. Cross-reference `crates/gateway/src/proxy.rs:284-366` for the existing pattern.

### C2. Cache claim is unverified — and currently false in this repo

- **Where:** plan.md "Cache Interaction (decided)"; phase-02 §4 "Caching consideration"; researcher-02 §9.
- **Problem:** Plan asserts "Upstream/origin cache stores the unredacted body … every client response — cache hit or miss — runs through redaction." This is repeated three times as a settled decision. **It is unsubstantiated for this codebase.** `crates/gateway/src/cache.rs` defines a moka cache, but `grep -rn "cache\.get\|CachedResponse" crates/gateway/src/proxy.rs` returns nothing. The cache is not wired into the Pingora response path at all, so claims about "post-cache" semantics are theoretical only. Pingora's *own* HTTP cache (researcher-02 §2) is also not enabled (no `cache_key_callback` / `cache_filter` overrides in `proxy.rs`). If a future phase wires either cache into the proxy without re-checking this assumption, redacted-vs-unredacted divergence will leak.
- **Suggested fix:** Either (a) downgrade the language to "if/when caching is enabled, …" and add an explicit follow-up TODO to verify when cache integration lands; or (b) add an explicit guard in `response_body_filter` that asserts `ctx.from_cache == false` and warns if a cached path ever bypasses it. The current "decided" language is overconfident given the actual state of `proxy.rs`.

### C3. Skip-flag default `None` means "redact" — wrong default for self-generated WAF responses

- **Where:** phase-02 §1 (`GatewayCtx` extension) + §3 (`response_filter` — Decide Skip Up-Front).
- **Problem:** Skip decision lives in `response_filter` (line 372 of current `proxy.rs`). But the existing WAF block path (`request_filter` at lines 256-263 and `request_body_filter` at lines 341-349) writes the response **directly via `session.write_response_*`** without ever entering `response_filter`. Result: when the WAF emits its own block page, neither `response_filter` nor `response_body_filter` runs — that's actually fine, no leak. **However**, if any future code path produces a synthesized response **after** `response_filter` has been bypassed (e.g. an upstream-error 502 generated by Pingora itself, an early `fail_to_proxy`, or a custom error response from a future feature), the default `body_redactor_skip_reason: None` and `response_body_buf: None` combine to: skip flag absent → look at `response_body_buf` → it's `None` → fast-return `Ok(())`. So actually the buffered-or-not check at `let Some(buf) = ctx.response_body_buf.as_mut() else { return Ok(None); };` does save us. **BUT** Phase 02 §3 also says "ctx.response_body_buf stays None — fast no-op path in body filter" — meaning the safety relies on `response_filter` ALWAYS running before any body chunk arrives. The trait does not guarantee this for self-generated responses. A defensive read of the documentation is required.
- **Suggested fix:** Make `body_redactor_done = true` the default in `GatewayCtx` (not `false`), and have `response_filter` flip it to `false` only when it has affirmatively decided to redact. That inverts the trust polarity to fail-safe. Document the invariant: "If `response_filter` did not run, do nothing — never assume redaction is needed without an explicit opt-in for this request."

## Major Concerns (should fix; can ship without if explicitly waived)

### M1. Compression skip is silent leak — operator has no choice mechanism

- **Where:** plan.md "Failure Mode = Fail-Open"; phase-02 §3 (`body_redactor_skip_reason`).
- **Problem:** Researcher-01 Trap 3 calls compression-skip a *known* leak. Researcher-02 §3 lists three options (skip / decompress / force identity). Plan unilaterally picks "skip + warn-log" with no operator override. A backend that returns `Content-Encoding: gzip` with `card_number` in the body will leak, and the operator can't even fail-close on it. The plan's own "Out of Scope" admits "decompress-recompress and `Accept-Encoding: identity` upstream injection are deferred." That's defensible, but the *config schema* should at least carry a forward-compatible enum (`compression_policy: skip | strip_encoding_request | block`) so we don't break the TOML when we add it.
- **Suggested fix:** Add `compression_policy: enum { Skip }` (single-variant enum today) to `BodyRedactorConfig` so future expansion is non-breaking. Or accept the breaking change later and call it out explicitly. At minimum, change the plan's warn-log to log on **every** compressed-and-skipped JSON response, not "single warn per request" (which Phase 02 §4 implies but never enforces). Operator monitoring only sees the leak surface if every event is recorded.

### M2. `case_insensitive=true` silently lower-cases operator-supplied `extra_fields`

- **Where:** phase-01 §2 (`from_config`) lines 156-158; plan.md "Walk Semantics" + Risk row "case_insensitive collides with operator-supplied".
- **Problem:** When `case_insensitive=true`, `extra_fields=["MyField"]` is folded to `myfield`. If the operator intended case-sensitive matching for that one field name (e.g. they have both `Token` (sensitive) and `token` (a JWT-payload type marker), they cannot express it. Worse: switching `case_insensitive` from `true` to `false` later silently re-enables what was hidden. The plan documents this in the Risk table as "operator-supplied names normalised the same way" — which states the behaviour but does not justify it.
- **Suggested fix:** Either (a) add a unit test specifically asserting the surprise behaviour (`extra_fields=["TokenFoo"]` with case_insensitive=true matches `tokenfoo`) so it's locked in — surprise becomes contract; or (b) keep `extra_fields` case-sensitive even when global `case_insensitive=true` (operator opts into precision when they extend). I lean (b) — the built-in catalog gets the loose mode, the operator override gets the strict mode. Flag for user decision.

### M3. Mask-as-string changes response schema for non-string fields

- **Where:** plan.md "Walk Semantics"; phase-01 unit test #15 (`mask_collapses_nested_value`); phase-01 unit test #17 (`null_or_number_value_masked`).
- **Problem:** A typed client (Swift, Kotlin, TS) consuming `{"secret": {"hash":"abc","salt":"xyz"}}` and expecting `secret: object` will hit a runtime decode error when the WAF replaces it with `"secret":"***REDACTED***"`. Same for `{"pin": 1234}` (number → string) and `{"locked": false}` (bool → string). This is documented as a feature in the plan ("regardless of original type"), but the *consequence* — every typed downstream that expects PCI-shaped fields breaks — is not surfaced to operators. PCI-DSS Req 3.4 cited in the plan does NOT require schema collapse; it only requires unreadability of the PAN.
- **Suggested fix:** Document loudly in `default.toml` comment block AND in `system-architecture.md` that v1 redaction collapses ALL value types to a string mask, so typed clients must accept a string everywhere a sensitive field is allowed. Optional follow-up: type-preserving mask (number → 0, bool → false, object → empty object, string → mask). Explicit user decision: ship type-collapse v1, or deliver type-preservation in the same PR? My read: ship type-collapse, but fail loudly in docs.

### M4. `body.take()` semantics — race with cache write-back / retry

- **Where:** phase-02 §4 (steal-and-replay logic).
- **Problem:** Plan correctly steals chunks via `body.take()` so Pingora doesn't double-emit. **However**, if Pingora's HTTP cache is ever turned on (see C2), the upstream-response-body filter runs *before* the body lands in cache. The plan uses the *post-cache* `response_body_filter`, so on the first request the body is fully captured into cache *unredacted* (good for the plan's stated semantics), and `body.take()` only affects the client-bound stream. **But:** if a hedge / retry / mirror is wired in (the codebase has `tunnel.rs` and `lb.rs` that hint at multi-leg paths), each leg may invoke the body filter chain again with a different chunk timeline. `ctx.body_redactor_done` is per-request — fine — but `body.take()` on a chunk that some later phase needs to read (e.g. a checksum trailer comparator) would hand it `None`. Researcher-02 §10 also notes `end_of_stream` may fire twice in HTTP/2; the plan handles double-fire via `body_redactor_done` (good), but does not address the dual case where `end_of_stream` never fires (request hangs forever because we buffer everything and never emit). That latter case is a denial-of-service against the *client*: WAF buffers full response, client times out, no bytes ever delivered.
- **Suggested fix:** Add a defensive fallback to phase-02: if buffer reaches `body_cap` AND `end_of_stream == false`, treat it the same as cap-overflow (drain, mark done, fail-open). This way a stuck stream cannot hang the client forever. Currently the plan only emits-on-overflow when the *next chunk would push past cap* — a stalled stream that buffers exactly at cap and stops sending will hang.

### M5. Test #5 specification is too vague for the chunk-stealing semantics

- **Where:** phase-03 §2 Test #5 ("multi-chunk JSON across boundaries — all bytes accounted for").
- **Problem:** "All bytes accounted for" is not a verifiable assertion. The real invariants the test must establish:
  1. The downstream stream sees `*body = None` for every non-final chunk (stolen).
  2. The downstream stream sees a single replacement chunk on `end_of_stream=true`.
  3. The bytes arriving at downstream are byte-equal to `serde_json::to_vec(&redacted_value)` — NOT to `chunk1 ++ chunk2`.
  4. `Content-Length` is either re-derived by Pingora or stripped (researcher-02 §4 says auto-handled, but the test must observe it).
  5. `body_redactor_done` is true after end_of_stream; subsequent invocations are no-ops.
- **Suggested fix:** Rewrite Test #5 with explicit assertions on: (a) the order of `*body` mutations across calls; (b) the byte-identity of the final chunk; (c) the `body_redactor_done` flag transition. Without these, the test passes trivially while the steal-and-replay path could be silently broken.

### M6. Strategy A test plan refers to non-existent precedent

- **Where:** phase-03 §2 Strategy A ("Mirror the test pattern used by existing `outbound::header_filter` tests if they exist").
- **Problem:** `crates/gateway/tests/` does not exist. FR-035 tests are *unit* tests inside `crates/waf-engine/src/outbound/header_filter.rs` — they test the redactor logic, not the Pingora hook integration. No precedent exists for how to drive `WafProxy::response_body_filter` from a test in this repo. The plan's "Mirror existing pattern" is ungrounded; the implementer will discover this on day 1 and fall back to Strategy B by default.
- **Suggested fix:** Drop the false precedent. Either commit Phase 03 to Strategy B (in-process Pingora server) — which is the only realistic option without scaffolding — or add a Phase-03 sub-task: "build the gateway-test harness." I'd argue building the harness is itself a separate effort that should not block FR-034. Simpler: keep Phase-03 unit tests in `body_redactor.rs` (covering parse-redact-serialise) and defer hook-level integration testing to a follow-up plan. The plan's currently-listed "10 integration test cases" are ambitious for the scaffolding cost.

### M7. Skip-on-non-JSON happens per-request in `response_filter` — but `response_filter` may be invoked for cached responses

- **Where:** phase-02 §3.
- **Problem:** Pingora's `response_filter` runs for cache hits AND fresh responses (researcher-02 §2 table). Good. **But** the only check we run is `Content-Type` and `Content-Encoding`. If the cache stores an unredacted JSON body and the cache-hit path serves it with a **stripped** Content-Type (some cache implementations don't preserve all headers), the redactor will skip silently and the body leaks. With the moka cache as currently defined (`headers: Vec<(String, String)>`), headers ARE preserved, but the cache-hit path through `response_filter` is not yet implemented (C2). Coupling the skip-decision to the upstream's content-type is brittle.
- **Suggested fix:** When (eventually) the cache is wired in, ensure the cached `Content-Type` is restored on `response_filter`'s `upstream_response` argument. Add an integration test that explicitly cycles through "miss → store → hit" and confirms redaction on both legs. This isn't blocking for FR-034 v1 but document it as a known follow-up in the plan's "Out of Scope" → "Future Work" section.

### M8. FR-033 boundary is hand-waved

- **Where:** plan.md "Out of Scope" line 130 + Detection Families table (`redact_pii`).
- **Problem:** Plan defines FR-033 as "value scanning for stack traces / API keys" and FR-034 as "field-name match." But `redact_pii` (`email`, `phone_number`, `dob`) is by-name catching of values that are *value-shaped*; FR-033's brief includes API keys (which are also value-shaped, often in fields named `api_key`). The two features will overlap on `api_key`-family fields. A future FR-033 implementer will either skip those fields (because FR-034 already covers them) or duplicate effort. The plan does not call this out as a decision the FR-033 author needs to make.
- **Suggested fix:** Add a note in the plan's "Out of Scope" / "FR-033 boundary" stating: "FR-034 catches sensitive fields by NAME; FR-033 will catch by VALUE. Any field caught by FR-034 is already redacted before FR-033 runs (post-cache, pre-client). FR-033 is scoped to fields whose names did NOT match FR-034 catalog." This pre-decides the overlap so FR-033 doesn't have to.

## Minor / Stylistic

- **Phase 04 commit message uses `docs:` prefix** for the third commit (TOML + docs). This is fine for `docs/`, but `configs/default.toml` is a *config-schema* change, not a docs change. Recommend `feat:` since the schema is part of the feature contract.
- **Phase 04 token-handling is mostly fine** but inconsistent: line 134 says "NEVER pasted into the command, NEVER committed" — good. Line 135 then says "type manually" — but `gh pr create` will record the env-var value into shell history if the user runs `export PRX_WAF_PAT=ghp_...` interactively without `set +o history`. Recommend explicitly `read -s -p "PAT: " PRX_WAF_PAT` (silent prompt, never enters history).
- **Phase-01 unit test #19** is misnamed: "body_cap_default_256kib" but `body_cap()` returns the configured cap, not the default. Test name should be `body_cap_returns_configured_value`.
- **Mask string `"***REDACTED***"`** is 14 chars — `Some(redacted_full)` will be smaller than original PAN values (16+ chars) but larger than e.g. `{"pin":1234}`. Pingora handles framing per researcher-02 §4 — fine. But a 256-KiB body fully redacted may blow past 256 KiB on output if every short int becomes a 14-char string. Output size is bounded by `original_size * (14 / 1)`, i.e. 14×. Phase-02's cap is on *input*, not output. Document.
- **`is_json_content_type`** intentionally rejects `text/json` (real legacy APIs use it — looking at you, IBM iSeries). Probably fine — niche enough — but worth a one-liner comment.
- **`HashSet<String>` for active fields** allocates per-name. Cheap and reasonable. Could be a `&'static [&'static str]` plus extras `HashSet<String>` — but YAGNI applies; HashSet is fine.
- **Plan "Standards & References" line 138** cites OWASP API3:2023 — that one is "Broken Object Property Level Authorization", which is *authz*, not *redaction*. The right OWASP reference for output sanitization is API8:2023 (Security Misconfiguration) or general OWASP Top 10 A04 (Insecure Design). Minor.
- **`from_config` returns `Option<Arc<Self>>`** — fine. But `if !cfg.enabled { return None; }` and then `if set.is_empty() { return None; }` together are subtle: an operator can have `enabled=true` with every family false and no extras and silently get a no-op redactor. Plan says "every family off → no-op redactor" intentionally. Fine, but should produce a `tracing::info!` once at startup so the operator sees they enabled the redactor for nothing.

## Things Done Well

- **KISS / YAGNI fully respected.** No JSONPath, no regex DSL, no per-route policy, no decompress-recompress. Each refused feature is named in "Out of Scope" with a short rationale.
- **FR-035 sibling pattern faithfully reused.** `OutboundConfig.body_redactor` mirrors `OutboundConfig.headers`; `WafProxy.body_redactor` mirrors `WafProxy.header_filter`; construction site in `crates/prx-waf/src/main.rs` is the right place (verified at line 1323).
- **Default-off master toggle.** `enabled: false` plus `redact_pii / redact_phi: false`-when-enabled is the correct conservative posture for shipping into existing deployments.
- **Idempotency guard (`body_redactor_done`)** correctly addresses researcher-02 §10's `end_of_stream`-may-fire-twice case.
- **256 KiB cap with operator override (`body_size_cap_bytes`)** — sane default, escape hatch present.
- **Hard cap on byte-size before parse** closes the JSON-bomb / deep-recursion vector; relies on serde_json's 128-deep default which is documented and safe.
- **Plan correctly escalates non-blocker decisions to user** via "User-Decided Open Questions" rather than silently picking.
- **Pre-edit `gitnexus_impact` instructions** in every phase respect this repo's `CLAUDE.md` non-negotiable.
- **Fail-open is correctly justified** as defense-in-depth — killing prod on parse error would be worse than the leak in degraded conditions. The reasoning is explicit in plan.md "Failure Mode = Fail-Open".
- **Test matrix in Phase-01 is comprehensive** (19 unit tests covering each family on/off, case modes, mask collapse, malformed input, cap, content-type accept/reject).
